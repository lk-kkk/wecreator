## 0. 需求来源与分析

### 0.1 附件需求摘要

附件《产品需求文档-项目及任务管理update(PRD)》提出了项目管理模块和任务管理模块的深化需求，核心要点：

1. **项目管理深化** — 项目进度看板（卡片+三色预警）、里程碑管理、项目阶段定义
2. **任务过程管控增强** — 检查点机制（进度/质量/风险）、工作日报、现场打卡、关键节点留痕
3. **交付物管理增强** — 版本管理、格式自检、验收流程标准化
4. **任务状态增强** — 三维状态（执行+验收+结算）、自动化机制（超时/风险/阻塞）
5. **沟通协作增强** — 任务级评论、问题上报（SLA 24h）、通知提醒体系
6. **数据分析** — 任务统计、进度分析、质量分析

### 0.3 设计原则

基于差异分析，本次需求遵循以下原则：

1. **不推翻现有架构** — 所有增强基于现有 V3.6.1 的数据模型和页面结构，采用"增量叠加"方式
2. **MVP 分层** — P0 纳入 Sprint 3（W13-W14），P1 纳入 Sprint 4（W15-W16），P2 纳入 Sprint 5
3. **复用优先** — 任务详情页三栏布局不变，通过新增 Tab 和组件扩展能力
4. **WeCreator 产品定位** — 我们是"创意零工撮合平台"而非通用项目管理工具，保持轻量化，不堆积重型 PM 功能

## 2. 项目管理增强（§4.9 扩展）

### 2.1 项目信息增强

在现有 §4.9.2 创建/编辑项目的 Modal 弹窗中，新增以下字段：

| 字段 | 类型 | 必填 | 说明 | V3.6已有 |
|------|------|------|------|---------|
| 项目名称 | 文本 | ✅ | 2-50字 | ✅ |
| 项目编号 | 自动生成 | — | PRJ-YYYYMMDD-NNN | ✅ |
| 客户所在地 | 城市选择 | ❌ | — | ✅ |
| 项目负责人 | 下拉 | ✅ | 默认当前用户 | ✅ |
| 项目描述 | 多行文本 | ❌ | 最多500字 | ✅ |
| **预期交付日期** | **日期** | **❌** | **项目整体期望完成日期（用于预警计算）** | 🔴 NEW |
| **项目阶段** | **单选** | **✅** | **需求确认 / 制作执行 / 验收交付（默认"需求确认"）** | 🔴 NEW |

**项目状态增强（4态替代3态）:**

```
V3.6 原始: 进行中 → 已完成 → 已归档（3态，单向）

V3.7 增强: 规划中 → 执行中 → 已完成 → 已归档（4态）
            │         │
            └─暂停─────┘  （规划中/执行中均可暂停，暂停可恢复）
```

| 状态 | 说明 | 允许操作 |
|------|------|---------|
| 规划中(planning) | 新建默认状态，尚未开始执行 | 编辑、暂停、流转到执行中 |
| 执行中(active) | 已有任务在执行 | 编辑、暂停、流转到已完成 |
| 暂停(suspended) | 临时暂停（不影响已关联任务执行） | 恢复到之前状态 |
| 已完成(completed) | 所有关联任务完成 | 流转到已归档 |
| 已归档(archived) | 终态，只读 | 无 |

> **V3.6 → V3.7 迁移:** 现有 `进行中` 映射为 `执行中(active)`，向后兼容。

**项目阶段自动流转规则:**
- 项目创建时默认阶段 = `需求确认`
- 项目下首个任务进入 `in_progress` 状态时，自动推进为 `制作执行`
- 项目下所有任务完成验收（`completed`），自动推进为 `验收交付`
- 手动流转优先于自动流转（PM 可随时调整阶段）

**三色预警机制:**

| 颜色 | 含义 | 触发条件 |
|------|------|---------|
| 🟢 绿色 | 正常 | 项目进度 ≥ 时间进度 × 80%（时间进度 = 已过天数 / 总工期天数） |
| 🟡 黄色 | 关注 | 项目进度 < 时间进度 × 80% 且 ≥ 时间进度 × 50% |
| 🔴 红色 | 延期风险 | 项目进度 < 时间进度 × 50%，或已超过预期交付日期 |

**项目进度计算方式:**
```
项目进度 = 关联任务中 status=completed 的数量 / 关联任务总数 × 100%
```

## 3. 任务管理增强

### 3.1 任务信息增强（§4.2.1 Step2 扩展）

在现有 Step2 表单中新增以下字段：

| 字段 | 类型 | 必填 | 说明 | 位置 |
|------|------|------|------|------|
| **任务编号** | 自动生成 | — | 格式 `TSK-YYYYMMDD-NNN`，创建草稿时生成，全局唯一 | 表单顶部只读展示 |
| **优先级** | 单选 | ✅ | P0 紧急 / P1 重要 / P2 常规，默认 P2 | "任务名称"字段下方 |
| **验收标准** | 富文本 | ❌ | 明确的量化指标和质量要求，最多1000字 | "总体交付目标"字段下方 |

**优先级定义:**

| 级别 | 含义 | 视觉标记 | 排序权重 |
|------|------|---------|---------|
| P0 紧急 | 阻塞性任务，需要立即处理 | 🔴 红色标签 | 最高 |
| P1 重要 | 核心功能，有明确时间节点 | 🟡 橙色标签 | 中 |
| P2 常规 | 常规需求，可正常排期 | 🔵 蓝色标签 | 最低 |

**任务编号生成规则:**
- 格式: `TSK-YYYYMMDD-NNN`（如 `TSK-20260418-001`）
- NNN 为当日全局递增序号（Redis INCR `task:seq:YYYYMMDD`）
- 草稿创建时即分配编号，不可修改

### 3.2 检查点管理（§4.5 新增子节）

> **设计定位:** 检查点是任务级别的过程管控手段，由企业 PM 在任务详情页设置。检查点不等于里程碑（里程碑是项目级别的）。

**检查点入口:** 任务详情页中栏（角色进度区域）底部新增「📋 检查点」可展开面板

**检查点类型:**

| 类型 | 用途 | 默认设置 |
|------|------|---------|
| 进度检查 | 核查任务执行进度是否符合预期 | 无默认，手动添加 |
| 质量卡点 | 关键质量节点的审核（如设计方案确认、技术方案确认） | 无默认，手动添加 |

**检查点数据结构:**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 检查点名称 | 文本 | ✅ | 2-50字 |
| 检查类型 | 单选 | ✅ | 进度检查 / 质量卡点 |
| 计划日期 | 日期 | ✅ | — |
| 验收人 | 下拉 | ✅ | 企业用户（默认任务创建人） |
| 说明 | 文本 | ❌ | 检查要求说明，最多200字 |

**检查点状态流转:**

```
待检查(pending) → 已提交(submitted) → 通过(passed) / 不通过(rejected)
                                              ↓
                                        需修改(revision) → 已提交 → ...
```

**操作规则:**
- 零工端在检查点到期前提交检查内容（文字说明 + 附件）
- 企业端验收人审核：通过 / 不通过（需填写不通过原因）
- 不通过后零工修改再提交，最多循环 3 次
- 检查点到期未提交 → 系统通知双方，标记为"已逾期"
- 每个任务最多 20 个检查点

**在任务详情页的展示:**

```
📋 检查点 (2/5)                              [+ 添加检查点]
┌────────────────────────────────────────────────────┐
│ ✅ 需求对齐会议    04-16  进度检查  已通过          │
│ ✅ 设计方案评审    04-20  质量卡点  已通过          │
│ ⏳ 中期进度检查    04-25  进度检查  待检查  剩2天   │
│ ⭕ 交付物初审     04-28  质量卡点  待检查          │
│ ⭕ 最终验收       05-01  质量卡点  待检查          │
└────────────────────────────────────────────────────┘
```

### 3.5 任务状态增强

**现有状态（V3.6.1）:**
```
tasks.status: draft/pending_review/published/in_progress/reviewing/completed/closed（7态）
```

**V3.7 保持任务主状态不变**，新增两个维度状态字段用于细粒度跟踪：

| 字段 | 类型 | 说明 |
|------|------|------|
| risk_level | ENUM NULL | 风险等级: green/yellow/red（系统自动计算，可手动覆盖） |
| acceptance_status | ENUM NULL | 验收汇总状态: pending/partial/all_passed（基于各角色验收情况汇总） |

**risk_level 自动计算规则:**

| 风险等级 | 触发条件 | 视觉标记 |
|---------|---------|---------|
| 🟢 green | 正常（默认） | 绿色图标 |
| 🟡 yellow | 距截止日 ≤ 5天 且 overall_progress < 70% | 黄色图标 + 状态栏提醒 |
| 🔴 red | 距截止日 ≤ 2天 且 overall_progress < 80%，或已超截止日期 | 红色图标 + 站内通知 |

**acceptance_status 计算规则:**
- `pending`: 无角色进入验收
- `partial`: 部分角色验收通过
- `all_passed`: 所有角色均验收通过（触发任务状态自动流转为 completed）

## 12. 非功能性需求

### 12.1 性能要求
- 项目看板加载时间 < 2秒（100 个项目以内）
- 评论列表加载 < 1秒（单任务 500 条评论以内）
- 通知列表加载 < 1秒
- 文件上传支持单文件最大 500MB（大型设计源文件/视频/3D模型）
- 风险等级计算 cron 每小时执行一次，耗时 < 30秒

### 12.2 兼容性
- 项目看板在 1280px / 1440px / 1920px 三种分辨率下自适应卡片列数（2/3/4列）
- 通知中心支持 WebSocket 实时推送（降级为轮询，间隔 30秒）
