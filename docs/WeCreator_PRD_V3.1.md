## 1. 产品概述

### 1.1 产品定位

**WeCreator 灵活用工撮合平台**是连接企业用工需求与零工人员的数字化中台。

**核心价值主张:**
- **对企业:** 快速组建灵活团队,按需付费,降低用人风险
- **对零工:** 获得高质量兼职收入,积累专业履历,建立个人品牌
- **对平台:** 收取服务费(8%),沉淀交易数据,构建信用体系护城河

企业通过 PC 后台发布任务包(含角色配置),平台完成智能撮合,零工通过微信小程序完成接单、执行、交付及收款的全流程闭环。

### 1.2 核心业务流程

> **V3.0 变更:** 结算流程从"直接微信零钱"改为"经第三方合规通道结算"。

```
企业充值预存款(发布前必须)
        ↓
企业发布任务包(配置角色岗位)
        ↓
系统预锁定资金(总预算 × 108%)
        ↓
平台撮合 / 企业定向分配
        ↓
零工接受角色任务(签署电子协议)
        ↓
零工执行任务(提交进度 & 交付物)
        ↓
企业验收(通过 / 退回修改,最多 2 次)
        ↓

        ↓

        ↓
解锁剩余资金(如有浮动)
        ↓
双向评价 → 信用数据沉淀
```


#### 4.2.6 任务检查点系统 ✨V3.7 NEW

> **V3.7 新增章节:** 检查点是企业对任务执行过程中的关键节点设置的质量关卡和进度检查。支持两种类型:进度检查(progress_check)和质量关卡(quality_gate)。

**检查点类型:**

| 类型 | 编码 | 说明 |
|------|------|------|
| 进度检查 | `progress_check` | 定期检查任务执行进度,确认是否按计划推进 |
| 质量关卡 | `quality_gate` | 关键里程碑节点的质量审核,需要提交成果并经审核通过 |

**状态流转:**

```
pending(待提交) → submitted(已提交) → passed(审核通过)
                                      → rejected(审核驳回) → submitted(重新提交)
pending → overdue(逾期,系统自动标记)
```

**创建检查点:**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 名称 | 文本 | ✅ | 最多50字 |
| 类型 | 选择 | ✅ | progress_check / quality_gate |
| 计划日期 | 日期 | ✅ | 检查点截止日期 |
| 审核人 | 下拉 | ✅ | 企业用户(company_users.id) |
| 说明 | 多行文本 | ❌ | 最多200字 |
| 排序 | 数字 | ✅ | 检查点顺序 |

**提交检查点(零工/企业):**

| 字段 | 类型 | 说明 |
|------|------|------|
| 提交内容 | 多行文本 | 检查点完成情况描述 |
| 附件 | 文件数组 | 提交的证据/成果URL数组 |

**审核检查点(企业端):**

| 字段 | 类型 | 说明 |
|------|------|------|
| 审核结果 | 选择 | passed(通过) / rejected(驳回) |
| 审核意见 | 多行文本 | 审核说明,驳回时必填 |

**SLA 规则:**
- 检查点超过计划日期未提交,系统自动标记为 `overdue`
- 驳回后 revisionCount +1,支持多次重新提交

**业务规则:**
- 检查点属于任务级别(不是角色级别)
- 仅企业端(super_admin/task_admin)可创建和审核检查点
- 检查点按 sortOrder 排序展示

> **设计说明:** 当企业使用私有部署模型、内部API网关或不兼容OpenAI格式的第三方服务时,需要完全自定义接口的请求/响应格式。选择Provider=「自定义」后,表单展开以下高级配置区域。

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 接口协议 | 单选 | ✅ | `openai_compatible`(兼容OpenAI Chat Completion格式) / `custom_http`(完全自定义HTTP接口) |

| Chat 路径 | URL | ✅ | 对话接口完整URL,如 `https://my-llm.internal/v1/chat` |

| 响应提取路径 | JSON Path | 条件 | 接口协议=custom_http时必填;指定从响应JSON中提取AI回复内容的路径,如 `data.choices[0].message.content` 或 `result.text` |

**接口协议说明:**

| 协议 | 适用场景 | 配置复杂度 |
|------|----------|------------|
| `openai_compatible` | 兼容OpenAI API格式的服务(如vLLM、Ollama、LiteLLM、FastChat、各云厂商OpenAI兼容接口) | 低 - 只需填Base URL+API Key+模型名 |
| `custom_http` | 完全非标准接口(企业内部AI网关、自研模型服务、特殊协议的三方服务) | 高 - 需配置请求模板+响应提取路径 |

**请求体模板示例(自定义HTTP):**

```json
// 变量插值: {{model}} {{messages}} {{temperature}} {{max_tokens}}
// messages 格式固定为 [{"role":"system","content":"..."},{"role":"user","content":"..."}]
{
  "model_name": "{{model}}",
  "input": {
    "messages": {{messages}}
  },
  "parameters": {
    "temperature": {{temperature}},
    "max_output_tokens": {{max_tokens}}
  }
}
```

**响应提取路径示例:**
- OpenAI标准: `choices[0].message.content`

**前端交互:**

**「测试连接」功能:**
- 点击后发送测试消息到LLM API
- 成功:绿色提示「✅ 连接成功,模型响应正常」
- 失败:红色提示具体错误信息(API Key无效/Endpoint不可达/模型不存在)

**调用统计(只读):** 本月总调用次数、本月Token消耗、估算费用

**安全规则:**
- API Key AES-256加密存储(复用现有加密方案 §7.3)

#### 5.3.3 人天模式结算规则 ✨V3.1 新增

> **V3.1 新增说明:** V3.0 对人天模式的结算触发时机、资金流、工时超出处理等关键规则未定义,此处补全。

**结算触发方式(按周结算):**

```
零工每日打卡 + 提交工作日志
    ↓
企业确认工时(T+1 24:00 前,逾期自动确认)
    ↓
每周日 23:59 系统自动生成周结算单
  · 汇总本周已确认工时:已确认天数 × 日薪
  · 生成结算明细(字段:日期、确认状态、单日金额)
    ↓
经第三方合规通道结算 → T+1 工作日到账零工微信零钱
    ↓
系统从锁定资金中扣除已结算金额
```

**资金锁定计算公式(人天模式):**

```
锁定金额 = 日薪 × 预估工作天数 × 需求人数 × 108%

示例:
  产品经理,¥800/天 × 12天 × 2人 = ¥19,200
  锁定金额 = ¥19,200 × 108% = ¥20,736
```

**工时超出处理规则:**

| 场景 | 处理方式 |
|------|----------|
| 实际工作天数 ≤ 预估天数 | 按实际确认天数结算,剩余锁定资金解锁退回企业账户 |
| 实际工作天数 > 预估天数 × 90% | 系统提前通知企业「预算即将用尽,请及时追加」 |
| 实际工作天数 > 预估天数 | 暂停结算,企业需「追加预算」(补充锁定资金)后继续;零工可继续打卡但工时记为「待确认」 |
| 企业主动结束人天任务 | 按已确认工时结算,剩余解锁;未确认部分自动确认后结算 |
| 零工主动退出 | 已确认工时结算,未确认工时不结算;记录一次「主动退出」 |

**人天模式任务自然结束规则:**

- 任务截止日期当天 23:59 自动触发最终结算
- 系统汇总所有已确认工时 → 生成最终结算单 → 解锁剩余资金
- 企业可在结束后 7 天内进行评价

#### 5.4.5 信用分体系 ← V3.3 新增

**计算公式:**

```
信用分 = 平均评分 × 40% + 完成率 × 30% + 按时交付率 × 20% + (1 - 争议率) × 10%
```

**各维度定义:**

| 维度 | 计算方式 | 满分 | 权重 |
|------|---------|------|------|
| 平均评分 | `workers.avg_rating / 5 × 100` | 100 | 40% |
| 完成率 | `workers.completion_rate × 100` | 100 | 30% |
| 按时交付率 | 在截止日期前提交验收的任务数 / 已完成任务总数 × 100 | 100 | 20% |
| 无争议率 | `(1 - 争议任务数 / 已完成任务总数) × 100` | 100 | 10% |

**信用分等级映射:**

| 分数范围 | 等级 | 颜色 | 标记 |
|---------|------|------|------|
| 90-100 | 极佳 | #52c41a 绿色 | ⭐⭐⭐⭐⭐ |
| 75-89 | 良好 | #1677ff 蓝色 | ⭐⭐⭐⭐ |
| 60-74 | 一般 | #faad14 橙色 | ⭐⭐⭐ |
| 40-59 | 较低 | #ff7a45 橙红 | ⭐⭐ |
| 0-39 | 待改善 | #ff4d4f 红色 | ⭐ |

**业务规则:**
- 新用户默认信用分 80 分(尚无历史数据时给予「良好」初始分)
- 信用分每日零点重新计算(定时任务)

#### 5.6.5 零工申请流程 ← V3.4 重写

> **V3.4 核心变更:** 企业确认后**直接创建 `role_assignments(status=accepted)`**,零工无需再次确认,立即进入执行状态。婩拒不改变任务状态,仅双端记录申请历史。

```
零工在服务广场点击某角色的「申请」按钮
    ↓
未实名 → 强制跳转实名认证
    ↓
弹出申请表单:
  · 自动填充当前角色档案(技能标签 + 经验年限)
  · 一句话自我介绍(必填,20-200字)
  · 期望报酬(选填,默认按角色预算显示)
  · 可开始时间(选填)
    ↓
点击「提交申请」
    ↓
系统生成申请记录(status=pending)
    ↓
企业 PC 后台收到申请通知(任务详情页 → 申请管理 Tab)
    ↓
企业审核:
  「✅ 确认」→ 自动创建 role_assignments(status=accepted)    ← V3.4 直接执行
             → 任务状态自动流转为 in_progress(如果当前为 published)
             → 通知零工「您的申请已通过,请开始执行任务」
```

## 6. 通用模块设计

### 6.1 撮合引擎(MVP 简版)

#### 6.1.1 定向分配(主要模式)

```
企业从零工库选择候选人 → 系统依次校验:

✅ 校验 1:零工已完成实名认证?
✅ 校验 2:零工账号状态正常(未被封禁)?
✅ 校验 3:零工当前时间段无冲突任务(时间重叠检测)?
✅ 校验 4:零工技能标签满足角色「必备技能」要求(软校验,不满足有警告但可强制分配)?

全部通过 → 发送任务邀约通知(站内消息 + 微信订阅消息)
任意不通过 → 给出明确提示,建议企业更换候选人

零工收到通知 → 24h 内响应(接受 / 婉拒)
超时未响应 → 系统标记为「自动婉拒」→ 通知企业可重新选择
```

#### 6.1.2 平台推荐(辅助模式)

**触发场景:** 企业选择「平台推荐」撮合方式,或定向分配后零工婉拒需重新选人。

**评分算法(MVP 简版,规则匹配):**

```python
def calculate_match_score(worker, role_slot):
    """
    满分 100 分,综合评估零工与岗位的匹配程度
    """
    score = 0

    # 1. 技能匹配度(30 分)
    required_skills = set(role_slot.required_skills)
    worker_skills = set(worker.skill_tags)
    if required_skills:
        hit_rate = len(required_skills & worker_skills) / len(required_skills)
        score += hit_rate * 25
    bonus_skills = set(role_slot.bonus_skills)
    if bonus_skills:
        bonus_hit = len(bonus_skills & worker_skills) / len(bonus_skills)
        score += bonus_hit * 5

    # 2. 同角色历史经验(25 分)
    same_role_count = worker.get_completed_count_by_role(role_slot.role_type)
    score += min(same_role_count * 5, 25)

    # 3. 综合完成率(20 分)
    score += worker.completion_rate * 20

    # 4. 平台综合评分(15 分)
    score += (worker.avg_rating / 5.0) * 15

    # 5. 响应速度(10 分)
    response_score = max(0, 1 - worker.avg_response_hours / 24) * 10
    score += response_score

    return round(score, 2)
```

**推荐结果展示(企业侧):**
- 推荐最多 **8 名**零工
- 展示:匹配分 / 同类角色完成单数 / 综合评分 / 认证状态
- 企业可展开查看零工完整主页
- 支持「刷新推荐」获取不同候选人

### 6.3 评价体系

#### 6.3.1 企业对零工评价

**触发:** 验收通过后弹出评价引导,**7 天内有效**(逾期不可补评)

| 维度 | 权重 | 打分方式 |
|------|------|------|
| 交付质量 | 35% | 1-5 星 |
| 准时程度 | 25% | 1-5 星 |
| 沟通效率 | 20% | 1-5 星 |
| 专业态度 | 20% | 1-5 星 |

- 综合评分 = 各维度加权平均
- 文字评价(选填,20-200 字)
- 正向标签快选:准时交付 / 沟通顺畅 / 质量超预期 / 专业靠谱 / 响应及时 / 值得推荐

#### 6.3.2 零工对企业评价

| 维度 | 权重 | 打分方式 |
|------|------|------|
| 需求清晰度 | 30% | 1-5 星 |
| 付款及时性 | 30% | 1-5 星 |
| 沟通友好度 | 40% | 1-5 星 |

文字评价(选填,20-200 字)+ 企业端无法看到(匿名保护)

#### 6.3.3 评价规则

- 双方均完成评价后,评价**互相可见**(防止策略性评价)
- 任意一方 7 天内未评价 → 对方评价**直接公开**
- 评价提交后**不可修改**
- 恶意评价申诉:被评价方可提交申诉(附证明材料),平台 3 个工作日内裁决
- **综合评分算法:** 近 50 条评价的加权均值(滚动窗口),权重随时间递减

### 6.4 争议仲裁

#### 6.4.1 争议发起条件

| 触发场景 | 可发起方 | 发起时限 |
|---------|---------|---------|
| 企业已进行 2 次退回仍不通过 | 任意一方 | 第 2 次退回后 72h 内 |
| 企业超过 3 工作日未验收(自动通过前可发起) | 零工 | 截止前 6h 内 |
| 零工认为退回原因不合理 | 零工 | 收到退回通知后 24h 内 |
| 企业认为零工提交内容与要求严重不符 | 企业 | 验收期内 |
| 零工失联超 72h | 企业 | 确认失联后 |

#### 6.4.2 仲裁流程(MVP 阶段:平台人工仲裁)

```
任意一方在争议入口发起申请
    ↓
填写争议说明(必填,100-1000 字)
上传证据材料(进度记录截图、沟通记录等,最多 10 个文件)
    ↓
系统生成争议工单,通知双方「争议处理中,请等待平台介入」
    ↓
平台仲裁员(运营人员)介入:
  - 查看双方提供的证据
  - 查看系统记录(进度更新时间戳、消息记录、文件上传记录)
  - 必要时联系双方电话沟通
    ↓
平台出具裁决(3 个工作日内):
  1 支持全额结算(交付物合格,按合同支付)
  2 支持部分结算(交付物部分合格,按比例)
  3 支持退款(交付物不合格,退回锁定资金给企业)
  4 双方协商解决(平台辅助,不强制裁决)
    ↓
双方可在 24h 内对裁决提出异议(仅限 1 次)
    ↓
异议受理后,升级至运营负责人复核
    ↓
最终裁决执行(系统自动处理资金)
```

**仲裁基本原则:**
- 以合同条款和系统记录为主要证据
- 进度更新时间戳和文件上传记录具有较强证明效力
- 鼓励双方协商,仲裁为最后手段
- 恶意发起争议(3 次以上经仲裁均判定为无效申请):扣除信用分

### 7.6 税务合规方案 ✨V3.0 新增

### 7.7 资金并发安全方案 ✨V3.1 新增

> **V3.1 新增说明:** V3.0 未对 `companies.balance` 和 `wallets.available_balance` 的并发访问提供安全方案,存在资金超卖/重复扣款风险。此处补全。

#### 7.7.1 企业账户余额安全(乐观锁 + 事务)

**核心场景:** 充值、锁定资金、解锁资金、结算扣款可能并发发生。

**方案:Prisma 乐观锁 + MySQL 事务:**

```typescript
// companies 表新增 version 字段(乐观锁)
// ALTER TABLE companies ADD COLUMN version INT DEFAULT 0;

async function lockFunds(companyId: bigint, amount: number) {
  return prisma.$transaction(async (tx) => {
    // 1. 读取当前余额 + 版本号
    const company = await tx.companies.findUnique({
      where: { id: companyId },
      select: { balance: true, locked_balance: true, version: true }
    });

    // 2. 业务校验
    if (company.balance < amount) {
      throw new Error('INSUFFICIENT_BALANCE');
    }

    // 3. 乐观锁更新(version 不匹配则失败)
    const updated = await tx.companies.updateMany({
      where: { id: companyId, version: company.version },
      data: {
        balance: { decrement: amount },
        locked_balance: { increment: amount },
        version: { increment: 1 }
      }
    });

    if (updated.count === 0) {
      throw new Error('CONCURRENT_CONFLICT'); // 重试
    }

    // 4. 记录流水
    await tx.financial_transactions.create({ ... });
  });
}
```

**重试策略:** 乐观锁冲突时自动重试,最多 3 次,每次间隔 100-300ms 随机退避。

#### 7.7.2 零工钱包余额安全(Redis 分布式锁)

**核心场景:** 结算入账和提现可能并发发生。

```
提现操作:
1. 获取 Redis 分布式锁:lock:wallet:{worker_id},超时 10s
2. 查询 wallets.available_balance
3. 校验余额 ≥ 提现金额
4. 扣减余额(MySQL 事务内)
5. 调用合规通道转账 API
6. 根据结果更新流水状态
7. 释放 Redis 锁

失败回滚:转账失败时,自动回滚余额扣减,写入失败流水记录。
```

#### 7.7.3 关键安全规则

| 规则 | 说明 |
|------|------|
| 金额二次校验 | 结算金额在 Task Service 和 Pay Service 分别独立计算,结果一致方可执行 |
| 幂等性保护 | 每笔结算/提现操作携带唯一 request_id,重复请求自动幂等 |
| 单笔人工审核 | 单笔结算金额 > ¥10,000 触发人工审核(飞书通知) |
| 每日提现限额 | 单笔 ≤ ¥5,000,每日 ≤ 3次,每日累计 ≤ ¥5,000 |
| 对账机制 | 每日凌晨 3:00 自动对账:企业余额 + 锁定余额 + 已结算金额 = 充值总额;异常时报警 |
| 异常监控 | 同一企业 1 小时内锁定 > 5 次触发报警;同一零工 1 小时内提现 > 3 次触发报警 |

#### 表:role_assignments(角色分配)✨ V3.1 修复

> **V3.1 修复说明:** V3.0 未处理 `task_roles.headcount > 1` 时的多人分配场景。新增 `slot_index` 字段和唯一索引,支持同一角色岗位分配多人。

| 字段 | 类型 | 说明 |
|------|------|------|

| worker_id | BIGINT FK | 关联零工 |
| status | ENUM | invited/accepted/rejected/expired/withdrawn |
| invited_at | TIMESTAMP | 邀约时间 |

| signed_ip | VARCHAR(45) NULL | 签署 IP |
| signed_ua | TEXT NULL | 签署设备 UA |
| UNIQUE INDEX | (task_role_id, slot_index) | 同一岗位每个槽位只能有一个有效分配 ✨V3.1 |
| INDEX | (worker_id, status) | 零工查询自己的任务分配 ✨V3.1 |

**多人岗位分配逻辑:**
- 任务发布时,系统根据 `headcount` 自动创建对应数量的空槽位(slot_index = 1, 2, ...N)
- 企业指定人员时,分配到下一个空闲槽位
- 角色岗位状态判定:所有槽位均为 `accepted` 时,角色状态变为 `assigned`
- 每个槽位独立跟踪进度、交付物、验收、结算

## 11. 接口规范概览

### 11.1 基础规范

| 项目 | 规范 |
|------|------|
| 协议 | HTTPS RESTful API |
| 数据格式 | JSON(Content-Type: application/json) |

| 时间格式 | ISO 8601,如 `2026-04-11T10:30:00+08:00` |
| 分页 | `?page=1&page_size=20`(默认 20,最大 100) |
| 版本控制 | URL 路径:`/api/v1/...` |

### 11.2 统一响应结构

```json
{
  "code": 200,
  "message": "success",
  "data": { ... },
  "request_id": "req_abc123",
  "timestamp": 1712806200000
}
```

**错误码:**

| 码 | 含义 |
|----|------|
| 200 | 成功 |
| 400 | 请求参数错误 |
| 401 | 未登录或 Token 失效 |
| 403 | 无权限 |
| 404 | 资源不存在 |
| 409 | 状态冲突 |
| 429 | 请求频率超限 |
| 500 | 服务器内部错误 |

## 16. 术语表

| 术语 | 定义 |
|------|------|
| **任务包** | 以整体交付成果为单位的任务形式,按交付物验收和结算 |
| **人天模式** | 以工作天数为计量单位的任务形式,按确认工时结算 |
| **角色岗位(TaskRole)** | 任务包中定义的具体工作岗位,如"产品经理(中级)",是撮合的基本单位 |
| **角色档案(WorkerRole)** | 零工在平台上配置的可接单角色信息,包含等级、日薪范围、技能标签 |
| **定向分配** | 企业从自有零工库中指定特定人员承接某个角色岗位 |
| **平台推荐** | 平台根据匹配算法推荐符合条件的零工候选人供企业选择 |
| **资金锁定** | 任务发布时,系统从企业可用余额中冻结对应金额(总预算 × 108%),确保结算资金充足 |
| **服务费** | 平台对每笔结算收取的 8% 佣金 |
| **合规通道费** | 第三方灵工结算平台收取的费用(含个税代缴),约 6.5%-7% |
| **T+1 到账** | 验收通过后的下一个工作日,结算金额到达零工微信零钱 |

## 18. 平台运营管理后台 · 详细需求 ✨V3.2 NEW

> **V3.2 新增章节:** V3.1 中平台运营定义为「手工操作,无独立 UI」。随着 Sprint 1 核心闭环跑通,平台需要独立的运营管理后台来支撑企业审核、零工管理、任务监控、资金监管、争议仲裁等日常运营工作。

### 18.1 产品定位与目标

#### 18.1.1 产品定位

**平台运营管理后台**是 WeCreator 平台运营团队的内部管理工具,提供全平台视角的数据监控、业务审核、风控干预和争议仲裁能力。

**核心价值:**
- **运营效率:** 从数据库直操作 + 飞书流程 → 可视化后台一站式管理
- **风控能力:** 实时监控异常资金流、违规任务、可疑账户
- **仲裁公正:** 结构化证据展示 + 完整时间线回溯 + 裁决模板
- **数据驱动:** 全平台核心指标看板,辅助运营决策

## 19. AI 智能体系统 · 详细需求 ✨V3.6 NEW

> **V3.6 新增章节:** 为企业端引入大模型(LLM)调用能力,通过AI智能体辅助任务发布决策。本章定义AI系统的完整技术架构、安全约束和扩展规划。

### 19.1 系统架构

```
企业端PC前端
    ↓ POST /api/v1/tasks/ai-chat (消息+会话ID+智能体ID)
    ↓
NestJS AI Module
    ├── 1. 查询企业LLM配置(llm_configs表,解密API Key)
    ├── 2. 加载智能体System Prompt(ai_agents表)
    ├── 3. 拼装Prompt(System + 对话历史 + 用户消息)
    ├── 4. LLM Adapter 路由(根据 provider + custom_protocol 选择适配器)
    │       ├─ OpenAI Adapter      → OpenAI / DeepSeek / 智谱
    │       ├─ Claude Adapter      → Anthropic Messages API
    │       ├─ Azure Adapter       → Azure OpenAI
    │       ├─ QwenAdapter         → 通义千问 DashScope API
    │       ├─ OpenAI-Compat       → 兼容接口(vLLM/Ollama/LiteLLM/FastChat等)
    │       └─ CustomHTTP Adapter  → 完全自定义接口(企业私有部署/内部网关)
    │           │  → 读取 custom_request_template, 插值变量
    │           │  → 按 custom_auth_type 注入认证
    │           │  → 按 custom_headers 添加自定义请求头
    │           └─ → 按 custom_response_path 提取响应内容
    ├── 5. 解析响应,判断是否需要工具调用
    │       → 工具调用: 查询平台角色库 / 查询历史任务数据
    │       → 最终轮次: 要求输出结构化JSON
    ├── 6. 记录调用统计(Token消耗+次数)
    └── 7. 返回前端(流式SSE或完整响应)
```

### 19.2 LLM 适配器架构(自定义接口支持) ← V3.6 增强

> **设计原则:** 采用策略模式(Strategy Pattern),每种 LLM Provider 对应一个 Adapter 实现。所有 Adapter 实现统一的 `LLMAdapter` 接口,AI Module 根据企业配置自动路由。

**统一接口定义:**

```typescript
interface LLMAdapter {
  // 发送对话请求,返回AI回复文本
  chat(params: {
    model: string;
    messages: Array<{ role: 'system' | 'user' | 'assistant'; content: string }>;
    temperature?: number;
    maxTokens?: number;
  }): Promise<{ content: string; usage: { promptTokens: number; completionTokens: number } }>;

  // 测试连接是否可用
  testConnection(): Promise<{ ok: boolean; error?: string }>;
}
```

**各适配器职责:**

| 适配器 | 触发条件 | 职责 |
|---------|----------|------|
| `OpenAIAdapter` | provider = openai / deepseek / zhipu | 直接调用 OpenAI Chat Completion API |
| `ClaudeAdapter` | provider = claude | 调用 Anthropic Messages API,转换请求/响应格式 |
| `AzureAdapter` | provider = azure_openai | 调用 Azure OpenAI API(含 deployment 路由) |
| `QwenAdapter` | provider = qwen | 调用 DashScope API,转换请求/响应格式 |
| `OpenAICompatAdapter` | provider = custom AND custom_protocol = openai_compatible | 使用用户配置的 base_url,调用 OpenAI兼容格式 |
| `CustomHTTPAdapter` | provider = custom AND custom_protocol = custom_http | 读取请求模板插值变量,按配置的认证方式和响应提取路径处理 |

**CustomHTTPAdapter 详细流程:**

```
1. 读取 custom_request_template (JSON模板字符串)
2. 插值变量: {{model}} → 实际模型名
              {{messages}} → JSON数组
              {{temperature}} → 数值
              {{max_tokens}} → 数值
3. 根据 custom_auth_type 注入认证:
   - bearer: Header 「Authorization: Bearer <credential>」
   - header: Header 「<custom_auth_header>: <credential>」
   - query:  URL拼接 ?api_key=<credential>
   - none:   不添加认证
4. 添加 custom_headers (额外自定义请求头)
5. 发送 HTTP 请求到 custom_chat_url
6. 从响应JSON中按 custom_response_path 提取内容
   例: "data.choices[0].message.content" → _.get(response, path)
7. 返回统一格式 { content, usage }
   (usage 无法提取时按字符数估算)
```

**安全约束:**
- 自定义接口 URL 必须为 HTTPS(或企业内网IP `10.x` / `172.16-31.x` / `192.168.x`)
- **SSRF防护黑名单:** 禁止访问 `127.0.0.0/8`(loopback) / `169.254.0.0/16`(云元数据) / `0.0.0.0/8` / `::1`(IPv6 loopback);禁止目标端口为 `3306/6379/27017/5432`(数据库常见端口) ← V3.6 复盘补充
- 请求模板中禁止包含 `{{api_key}}` 等密钥变量(认证统一走 auth 配置)
- 响应超时 30s,防止恶意服务堵塞
- 自定义接口的调用同样纳入频率限制和Token统计

### 19.5 安全与限制
