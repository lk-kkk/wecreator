## 一、逻辑自洽性审查(Logic Consistency)

### 🔴 P0 — 阻断级问题(2 个)

#### P0-1: tasks 表 status 枚举与任务状态机不一致

| 位置 | 内容 |
|------|------|
| §10.2 tasks 表 | `draft / matching / in_progress / reviewing / completed / closed` (6态) |
| §3.1 功能树 / CLAUDE.md 状态机 | `draft → pending_review → published → in_progress → reviewing → completed → closed` (7态) |

**矛盾点:**
- 表中缺少 `pending_review`(待审核)和 `published`(已发布)两个状态
- 表中有 `matching` 状态，但状态机中没有
- 这是 V3.0→V3.1 演化过程中产生的历史不一致，不是 V3.6 引入的

**影响:** 后端开发时无法确定正确的状态枚举定义，前端 Tab 筛选器和状态标签体系会与数据库不匹配。

**建议修复:** 统一为 7 态 `draft / pending_review / published / in_progress / reviewing / completed / closed`，删除 `matching`。

---

#### P0-2: AI 生成的 `total_budget` 字段无处可填

| 位置 | 内容 |
|------|------|
| §4.2.1 AI 生成字段映射表 | 建议预算 → `total_budget` |
| §4.2.1 Step2 表单 | **没有 total_budget 字段** |
| §4.2.1 Step3 角色汇总 | total_budget = 各角色报酬之和，**由系统自动计算，不可手动修改** |

**矛盾点:**
- AI 生成了 `total_budget`(如 ¥35,000)，但 Step2 表单没有预算输入字段
- Step3 的总预算是角色报酬加总后自动计算的，明确标注"不可手动修改"
- 「一键填充到表单」时 total_budget 无法落地

**影响:** AI 最核心的产出之一(预算推荐)无法回填到表单，功能闭环断裂。

**建议修复(二选一):**
- **方案 A:** AI 的 `total_budget` 仅作为参考展示在对话结果卡片中，不回填表单。真正回填的是 `roles[]`(角色配置建议)到 Step3，由 Step3 自动算出总预算
- **方案 B:** 在 Step2 增加一个只读的"AI 建议预算"展示区域(非表单字段)，作为参考信息

### 🟡 P1 — 重要问题(5 个)

#### P1-1: V3.6 附件上传缺少专用 API 接口

| 已有 | 缺失 |
|------|------|
| `POST /api/v1/common/upload/presign` — OSS 直传预签名 | 附件元数据写入 `task_attachments` 的 API |
| `task_attachments` 数据表(§10.4) | 附件删除 API (`DELETE`) |
| V3.6 变更清单第 10 条称"新增 12 个 API" | 12 个 API 中没有附件相关接口 |

**分析:** 现有 presign 接口只负责获取 OSS 上传凭证。文件上传到 OSS 后，前端需要将 `file_name / file_url / file_size / file_type` 写入 `task_attachments` 表 —— 但没有对应的 API。

**建议补充 2 个接口:**
- `POST /api/v1/tasks/:id/attachments` — 上传完成后保存附件元数据
- `DELETE /api/v1/tasks/:id/attachments/:attachmentId` — 删除附件

更新 V3.6 接口总数从 12 → 14。

#### P1-2: AI 对话会话存储方案未定义

| 已定义 | 缺失 |
|--------|------|
| §19.3 对话协议(请求/响应 JSON 格式) | 对话会话(session)存储在哪？ |
| §4.2.1 "对话记录保存到任务草稿元数据" | 没有定义 AI 对话表 / 集合 |
| §19.5 "审计日志: 记录每次 AI 调用" | 审计日志存储在哪张表？ |

**分析:** 
- IM 消息已有 MongoDB 的 messages 集合 + MySQL 的 conversations 索引
- AI 对话可以复用同一模式(MongoDB 存消息体 + MySQL 存会话索引)
- 但需要明确定义 `ai_chat_sessions` 和 `ai_chat_messages` 的结构

**建议补充:**
- MySQL: `ai_chat_sessions` 表(session_id, company_id, user_id, agent_id, task_draft_id, message_count, created_at, updated_at)
- MongoDB: `ai_chat_messages` 集合(session_id, role, content, structured?, token_count, created_at)

#### P1-3: operator 角色「使用智能体=✅」但无使用场景

| 权限矩阵 | 实际场景 |
|----------|---------|
| operator: 使用智能体 = ✅ | AI 任务顾问入口仅在发布任务 Step2 |
| operator: 发布任务 = ❌ | operator 永远无法进入 Step2 页面 |

**分析:** 当前 V3.6 中智能体的唯一使用场景是发布任务 Step2 的 AI 任务顾问。operator 不能发布任务，所以即使有"使用智能体"权限也无法行使。

**建议(二选一):**
- **方案 A(保守):** 将 operator 的"使用智能体"改为 ❌，等未来有其他智能体使用场景时再开放
- **方案 B(前瞻):** 在 PRD 中明确标注"V3.6 智能体使用场景仅限发布任务，operator 权限预留供后续扩展(如零工推荐顾问、数据分析顾问)"

**问题:** `llm_configs` 和 `ai_agents` 表均有"每月重置"字段，但 PRD 未定义重置触发方式。

**影响:** 开发时不知道用什么方案：
- 定时任务(cron)每月 1 日凌晨重置？
- 写入时判断月份，跨月自动归零？
- 使用 Redis 月份前缀 key？

**建议补充:** 在 §19.5 安全与限制中增加一行：

| 规则 | 说明 |
|------|------|
| 月度统计重置 | 每月 1 日 00:00 由后端定时任务(cron)将 `monthly_call_count` 和 `monthly_token_count` 归零;重置前将当月数据写入 `llm_usage_monthly_log`(可选,用于历史统计);ai_agents 的 monthly_call_count 同步重置 |

### 🟢 P2 — 建议级问题(4 个)

#### P2-1: 项目不可删除 — 需确认是设计决策还是遗漏

§4.9.2 定义了项目状态单向流转(进行中→已完成→已归档)，操作列只有"编辑/归档"，没有"删除"。API 也没有 DELETE 接口。

**分析:** 如果是有意设计(如审计追溯需要)，建议在 PRD 中明确标注"项目不可删除，仅支持归档"。如果是遗漏，则需补充删除逻辑(无关联任务时可删，有关联任务时禁删)。

#### P2-2: 项目详情页缺少编辑入口

§4.9.3 项目详情页布局只展示了基本信息和关联任务列表，没有"编辑"按钮。用户需要返回项目列表才能编辑项目信息。

**建议:** 在项目详情页头部增加"编辑"图标按钮。

#### P2-3: AI 任务顾问对话中的关联项目预填

Step2 表单新增了"关联项目"字段，但 AI 任务顾问生成的结构化字段中没有 `project_id`。

**建议:** 如果用户在打开 AI 顾问前已选择了关联项目，AI 可以在对话中感知到项目上下文(如"这是品牌VI升级项目下的任务")，但这需要在 §19.3 请求格式的 `context` 中增加 `projectId` 字段。

#### P2-4: task_attachments 表缺少 INDEX

§10.4 定义了 task_attachments 表，标注了 `INDEX (task_id)`，但与其他表的 INDEX 定义方式不一致(其他表在 §10.5 索引规划中定义)。建议在 §10.5 中补充 task_attachments 的索引。

## 二、需求完整性审查(Completeness)

### 🔴 缺失的关键需求(3 个)

#### C-1: AI 对话异常处理 UX 未定义 [P1]

当前 §4.2.1 和 §19 只定义了正常对话流程。以下异常场景的用户体验未定义：

| 异常场景 | 缺失的 UX 定义 |
|----------|----------------|
| LLM API 超时(>30s) | 对话框显示什么？能否重试？ |
| LLM API 返回错误(500/429) | 提示语措辞？是否自动重试？ |
| 企业 LLM 配置被修改/删除(对话中) | 正在进行的对话如何处理？ |
| AI 返回的 JSON 格式错误 | 「一键填充」按钮是否禁用？如何提示？ |
| 网络断开 | 对话消息是否丢失？ |

**建议补充:** 在 §4.2.1 AI 任务顾问章节增加"异常处理"小节:
- 超时: 显示"AI 正在思考中..."loading 动画，30s 后显示"响应超时，请点击重试"
- API 错误: 显示"AI 暂时无法回复，请稍后再试" + 重试按钮
- JSON 解析失败: 显示 AI 文本回复，但「一键填充」按钮灰色不可点，提示"本次建议格式异常，请继续对话让 AI 重新生成"
- 网络断开: 未发送消息保留在输入框，已发送消息显示发送失败图标

#### C-2: 附件的预览/下载能力未定义 [P1]

§4.2.1 定义了附件上传，但未定义：
- 已上传附件在 Step2 表单中如何展示？(文件名+大小+类型图标——有提及)
- 附件能否预览？(PDF 在线预览？图片缩略图？)
- 附件能否下载？
- 任务发布后，零工端是否可查看/下载任务附件？

**建议补充:**
- 企业端: Step2 表单中显示已上传附件列表(文件名+大小+类型图标+删除按钮)，点击可下载
- 任务详情页: 展示附件列表，企业端可下载
- 零工端: 接单后可查看并下载任务附件(需在零工端任务详情接口中返回 attachments)
- 零工端 API: `GET /api/v1/worker/tasks/{id}` 响应中增加 `attachments[]` 字段

#### C-3: 自定义 HTTP 接口的 SSRF 防护未充分定义 [P1]

§19.2 提到"自定义接口 URL 必须为 HTTPS(或内网 IP 10.x/172.16-31.x/192.168.x)"。

**问题:** 这个规则本身有安全漏洞：
- 允许内网 IP 意味着可以访问企业内部任何服务(如 Redis、MySQL、元数据服务)
- 攻击者可以配置 `http://169.254.169.254/latest/meta-data/` 获取云服务器元数据
- 应该禁止访问 `127.0.0.1`、`localhost`、`169.254.x.x` 等

**建议补充黑名单:**
```
禁止访问:
- 127.0.0.0/8 (loopback)
- 169.254.0.0/16 (link-local / 云元数据)
- 0.0.0.0/8
- 已知的内部服务端口(3306/6379/27017)
```

## 三、技术可执行性审查(Executability)

### ⚠️ 需要补充技术方案的点(4 个)

#### E-1: LLM Adapter 策略模式实现方案 [P1]

PRD §19.2 定义了统一 `LLMAdapter` 接口和 6 种适配器，但缺少 NestJS 具体实现指引。

**建议技术方案:**
```
NestJS Module:
  @Module({
    providers: [
      LLMAdapterFactory,      // 工厂类，根据配置返回对应 Adapter
      OpenAIAdapter,
      ClaudeAdapter,
      AzureAdapter,
      QwenAdapter,
      OpenAICompatAdapter,
      CustomHTTPAdapter,
    ]
  })
```
- 使用 `LLMAdapterFactory.create(config: LlmConfig): LLMAdapter` 工厂方法
- CustomHTTPAdapter 中使用 `lodash.get()` 实现 JSON Path 提取
- 模板变量插值使用正则替换 `{{variable}}`，需做安全校验(防注入)

---

#### E-2: 多轮对话上下文管理 [P1]

§19.3 定义了 sessionId 保持多轮对话，但未定义上下文传递策略。

**潜在问题:** 30 轮对话全量发送给 LLM 可能超出 context window(如 gpt-4o 128K，但 30 轮长对话可达 50K+ tokens，费用高昂)。

**建议技术方案:**
- 前 10 轮: 全量发送
- 10-20 轮: 保留 System Prompt + 前 3 轮摘要 + 后 7 轮全量
- 20-30 轮: 保留 System Prompt + 全局摘要 + 后 5 轮全量
- 在 §19.3 中增加"上下文窗口管理策略"说明

---

#### E-3: 流式响应(SSE)的前后端实现 [P2]

§19.1 提到支持 SSE 流式返回，但 PRD 未定义是 MVP 必需还是可选。

**建议:** 明确标注为 **可选优化**(非 MVP 必需)。原因:
- SSE 需要 NestJS 的 `@Sse()` 装饰器 + Observable
- 前端需要 `EventSource` 或 `fetch` 流式读取
- CustomHTTPAdapter 的流式解析复杂度高(每种第三方流格式不同)
- MVP 阶段使用完整响应(loading 状态 + 一次性返回)，体验足够

---

#### E-4: 项目编号并发生成 [P2]

格式 `PRJ-YYYYMMDD-NNN`，NNN 需全局唯一。

**建议技术方案:**
- 使用 MySQL 自增序列 + 日期前缀
- 或 Redis INCR 计数器: `project:seq:20260418` → INCR → 格式化为 3 位
- 并发量低(每企业每天创建项目不超过几个)，MySQL 事务内 SELECT MAX + INSERT 也可接受
